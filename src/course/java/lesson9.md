# 継承を理解しよう
共通処理を親クラスにまとめ、子クラスで差分を足す「継承」と、<code>super</code>・オーバーライドの基本を押さえます。

## 本章の目標

本章では以下を目標にして学習します。

- 継承が解決しようとしている問題を説明できること
- <code>extends</code> で子クラスを定義できること
- <code>super</code>（コンストラクタ・メソッド）の役割を説明できること
- メソッドの**オーバーライド**（上書き）を行い、<code>@Override</code> を付けられること
- 継承の使いどころと**乱用の注意**を述べられること

## 1. 継承とは

**継承**は、既存のクラス（**親クラス・スーパークラス**）の性質を引き継ぎ、**子クラス・サブクラス**を定義する仕組みです。**共通部分を1か所**にまとめ、**差分だけ子に書く**のが目的です。

```java
public class Animal {
    String name;

    public Animal(String name) {
        this.name = name;
    }

    public void eat() {
        System.out.println(name + " が食べる");
    }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name);  // 親のコンストラクタ呼び出し
    }

    @Override
    public void eat() {
        System.out.println(name + " がドッグフードを食べる");
    }
}
```

```java
Dog d = new Dog("ポチ");
d.eat();
```

> **ポイント**
>
> Java は**単一継承**（<code>extends</code> は1クラスだけ）です。複数の型の振る舞いを組み合わせたい場合は、後の学習で**インターフェース**などが登場します。

## 2. extends と is-a の関係

「子は親の一種（is-a）」と言えるとき、継承が自然です。

- <code>Dog</code> **is a** <code>Animal</code> → <code>extends Animal</code> は自然
- 「四角形 is a 車」のような関係は無理に継承しない

> **注意**
>
> **コードの再利用だけ**が目的で継承すると、後から関係が破綻しやすいです。「型としての関係が正しいか」を先に考えましょう。

## 3. super（コンストラクタ）

子クラスのコンストラクタの**先頭**で、親のコンストラクタを <code>super(...)</code> で呼びます。省略した場合、**引数なしの <code>super()</code>** が暗黙に呼ばれます（親に引数なしコンストラクタがないとエラー）。

```java
public class Cat extends Animal {
    public Cat(String name) {
        super(name);
    }
}
```

## 4. メソッドのオーバーライド

子クラスで**同じシグネチャ**のメソッドを定義すると、親の実装を**上書き**します。意図を明確にするために <code>@Override</code> アノテーションを付けるのが一般的です。

```java
@Override
public void eat() {
    System.out.println(name + " がチュールを食べる");
}
```

> **ポイント**
>
> <code>@Override</code> を付けると、親にそのメソッドがないのに書いてしまった場合などに**コンパイル時に気づけます**。

## 5. protected と final（入門）

- <code>protected</code>：同じパッケージ＋**サブクラス**からアクセス可能（詳細はカプセル化の章とセットで）
- <code>final</code> クラス：**継承禁止**。<code>final</code> メソッド：**オーバーライド禁止**

```java
public final class Utility {
    // 継承させたくないユーティリティ群など
}
```

## 6. 継承より合成？（ひとこと）

「すべてを継承で表す」と深い階層や窮屈な親子になりがちです。**持つ（has-a）** で表現できるなら、**フィールドで別オブジェクトを持つ**（合成・コンポジション）の方が適切なことも多いです。まずは継承の構文を固め、設計判断は経験とセットで育てます。

## 7. まとめ

| テーマ | 要点 |
| --- | --- |
| 継承の目的 | 共通を親に集約し、子は差分を書く |
| <code>extends</code> | 親クラスを1つ指定 |
| <code>super(...)</code> | 親コンストラクタ呼び出し。子コンストラクタの流れの起点 |
| オーバーライド | 子で同じシグネチャのメソッドを上書き。<code>@Override</code> 推奨 |
| is-a | 型としての関係が自然なときに継承を検討 |
| 注意 | 再利用だけの継承、深すぎる階層は避ける |

> **ポイント**
>
> 継承は強力ですが、**親子の結びつきが強い**という側面もあります。次の多態性で「親型の変数で子を扱う」イメージを掴むと、継承の使いどころがより見えてきます。
