# カプセル化を理解しよう
内部の状態を隠し、必要な操作だけを公開する「カプセル化」と、<code>private</code>・ゲッター／セッター・不変条件の守り方を押さえます。

## 本章の目標

本章では以下を目標にして学習します。

- カプセル化の目的（なぜ隠すか）を説明できること
- <code>private</code> / <code>public</code> の違いと使いどころを説明できること
- **ゲッター／セッター**で安全に状態にアクセスする形を書けること
- セッターで**不正な値を弾く**（バリデーション）イメージを持てること
- パッケージプライベート（デフォルト）を**名前だけ**理解すること

## 1. カプセル化とは

**カプセル化**は、オブジェクトの**内部表現や状態を外から直接いじれないようにし**、**決められた操作（メソッド）だけ**を公開する考え方です。

| 効果 | 説明 |
| --- | --- |
| 変更に強い | 内部の持ち方を変えても、公開メソッドさえ守れば呼び出し側への影響を減らせる |
| 不変条件の維持 | 「残高が負にならない」などをクラス内で保証しやすい |
| 誤用の防止 | 好き勝手にフィールドを書き換えられない |

> **ポイント**
>
> 「全部 <code>public</code> フィールド」でも動きますが、**大きなコードベースでは破綻しやすい**です。フィールドは基本 <code>private</code> から始める習慣をつけましょう。

## 2. private と public

```java
public class BankAccount {
    private int balance;  // 外から直接触れない

    public BankAccount(int initial) {
        if (initial < 0) {
            throw new IllegalArgumentException("初期残高が負です");
        }
        this.balance = initial;
    }

    public int getBalance() {
        return balance;
    }

    public void deposit(int amount) {
        if (amount <= 0) {
            return;
        }
        balance += amount;
    }

    public boolean withdraw(int amount) {
        if (amount <= 0 || balance < amount) {
            return false;
        }
        balance -= amount;
        return true;
    }
}
```

- <code>private</code>：そのクラス**内部だけ**
- <code>public</code>：どこからでも（公開 API）

## 3. ゲッターとセッター

- **ゲッター**：状態を**読む**（<code>getBalance</code> など）
- **セッター**：状態を**書く**（<code>setXxx</code>）。必須ではなく、**必要なときだけ**公開します。

```java
public class Person {
    private String name;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("名前が不正");
        }
        this.name = name;
    }
}
```

> **ポイント**
>
> 「とりあえずすべてにセッター」も乱用しがちです。**本当に外から変えてよいか**を考え、変えたくないならセッターを出さない／不変オブジェクトにする、などが選べます。

## 4. 不変（immutable）のひとこと

生成後に状態が変わらないオブジェクトは**不変**と呼ばれ、並行処理やバグ防止の面で有利なことがあります。すべて <code>final</code> フィールド＋セッターなし、などが典型です（ここでは「選択肢がある」程度でOK）。

## 5. パッケージプライベート（デフォルト）

アクセス修飾子を何も付けないと、**同じパッケージ内**からだけ見える（パッケージプライベート）になります。小さなモジュール内でのみ共有したいクラスやメソッドに使うことがあります。

## 6. カプセル化と継承

サブクラスから親の <code>private</code> フィールドは**直接触れません**。必要なら親が <code>protected</code> のメソッドで操作を提供する、といった設計になります。継承とセットで「隠すべき境界」を考えるとよいです。

> **注意**
>
> <code>public</code> なゲッターで**内部のコレクションそのもの**を返すと、外から中身を書き換えられてカプセル化が破れることがあります。必要なら**コピーを返す**・**読み取り専用ビュー**を返す、などの工夫が現場では出てきます。

## 7. まとめ

| テーマ | 要点 |
| --- | --- |
| カプセル化 | 内部を隠し、公開 API で操作を制御する |
| <code>private</code> | フィールドは基本非公開 |
| ゲッター | 読み取り窓口。不要なら作らない |
| セッター | 書き込み窓口。バリデーションを入れることが多い |
| 不変 | 状態を変えない設計も有力な選択肢 |
| パッケージ | 修飾子なし＝同パッケージのみ（ざっくり） |

> **ポイント**
>
> カプセル化は「縛り」ではなく、**チームで長く保守するための投資**です。小さなクラスから <code>private</code> とメソッド経由の操作を試し、「どこが公開面か」を意識してみてください。ここまでの変数・制御構造・配列・メソッド・クラス・継承・多態性とつながるので、全体を振り返るのもおすすめです。
