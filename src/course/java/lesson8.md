# オブジェクト指向を理解しよう
オブジェクト指向が目指す考え方と、手続き型との違い、設計で使う基本用語を整理します。

## 本章の目標

本章では以下を目標にして学習します。

- オブジェクト指向で重視する考え方を自分の言葉で説明できること
- **オブジェクト**（状態＋振る舞い）のイメージを持てること
- クラス・インスタンス・メッセージ（メソッド呼び出し）の関係を説明できること
- 継承・多態性・カプセル化が**何のためにあるか**を述べられること

## 1. オブジェクト指向とは（ざっくり）

**オブジェクト指向（OOP）**は、プログラムを**オブジェクトの集合**として組み立てる考え方です。各オブジェクトは**データ（状態）**と**それを操作する手続き（振る舞い）**をまとめ持ちます。

| 考え方 | イメージ |
| --- | --- |
| オブジェクト | 現実の「モノ」や「役割」をソフトウェア上の単位にしたもの |
| クラス | オブジェクトの共通の型・設計図 |
| メッセージ | 他のオブジェクトに「お願い」する＝メソッドを呼ぶこと |

> **ポイント**
>
> 「データと処理がバラバラ」だと、変更の影響範囲が追いにくくなります。オブジェクトにまとめると、**責任の所在**がはっきりしやすくなります。

## 2. 手続き型との違い（感覚）

手続き型寄りの書き方は「上から手順を実行」が中心です。オブジェクト指向では**オブジェクトに任せる**ことが増えます。

```java
// 手続き寄り：データを外からいじるイメージ
int price = 1000;
int taxIncluded = (int) (price * 1.1);

// オブジェクト寄り：オブジェクトに任せるイメージ
// Price p = new Price(1000);
// int taxIncluded = p.withTax();  // 中で計算ルールを隠せる
```

どちらが常に正しい、ではなく、**規模が大きくなるほど**オブジェクトに責任を分けるメリットが出やすい、と理解しておくとよいです。

## 3. オブジェクト＝状態＋振る舞い

- **状態**：フィールド（今の値）
- **振る舞い**：メソッド（できること）

銀行口座なら、残高は状態、入金・出金は振る舞い、というように対応づけます。

> **ポイント**
>
> 「このデータは誰が責任を持って変えるべきか？」と考えると、クラス分割のヒントになります。

## 4. カプセル化・継承・多態性（全体像）

オブジェクト指向でよくセットで説明される3つです。

| 用語 | 一言で |
| --- | --- |
| **カプセル化** | 中身を隠し、必要な窓口だけ公開する（次章・lesson11 で深掘り） |
| **継承** | 共通部分を親クラスにまとめ、子が差分を足す（lesson9） |
| **多態性** | 同じ型の変数でも、実体によって振る舞いが変わる（lesson10） |

ここでは**名前と目的**を掴み、あとの章で手を動かして慣れていきましょう。

## 5. 設計で使う心構え（現場目線）

- **単一責任に近づける**：1クラスがやることが増えすぎたら分割を検討
- **依存を減らす**：知りすぎているクラスほど変更に弱い
- **名前で意図が伝わる**：クラス名・メソッド名はドキュメントの一部

> **ポイント**
>
> 設計は一度で完璧にならず、**動かしてから直す**ことも普通です。まずは小さなドメイン（例：図書館の本、ECのカート）をクラスに落としてみる練習がおすすめです。

## 6. まとめ

| テーマ | 要点 |
| --- | --- |
| OOP の狙い | 変更に強く、責任が追いやすい構造にする |
| オブジェクト | 状態（フィールド）と振る舞い（メソッド）の単位 |
| クラス | オブジェクトの設計図・型 |
| メッセージ | メソッド呼び出しで他オブジェクトに依頼 |
| 3大要素 | カプセル化・継承・多態性（以降の章で具体化） |
| 実践 | 小さな題材でクラスを切り、言葉に落とす練習 |

> **ポイント**
>
> 用語暗記より、「なぜ分けるのか」「誰が責任を持つか」を考える癖が、のちの設計学習の土台になります。次の継承・多態性・カプセル化の各章で、一つずつつなげていきましょう。
