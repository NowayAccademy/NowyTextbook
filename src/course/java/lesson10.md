# 多態性を理解しよう
親型の変数で子の実体を扱い、実行時に振る舞いが切り替わる「多態性」と、それを支える動的束縛のイメージを押さえます。

## 本章の目標

本章では以下を目標にして学習します。

- 多態性（ポリモーフィズム）を**自分の言葉**で説明できること
- **スーパータイプの変数にサブタイプのオブジェクト**を代入できることを理解すること
- オーバーライドしたメソッドが、**実際のオブジェクト型**に応じて呼ばれることを説明できること
- <code>instanceof</code> で型を安全に確認できること

## 1. 多態性とは

**多態性**は、「**同じ型（親型）として扱いながら、実体ごとに違う振る舞いをする**」性質です。Java では主に**継承とオーバーライド**と組み合わせて現れます。

```java
Animal a1 = new Dog("ポチ");
Animal a2 = new Cat("タマ");

a1.eat();  // Dog の eat
a2.eat();  // Cat の eat
```

変数の型は <code>Animal</code> ですが、**中身が Dog / Cat** なので、呼ばれる <code>eat</code> の中身が変わります。

> **ポイント**
>
> 「呼び出し側は <code>Animal</code> としてシンプルに書く」「具体的な種類は実体側が決める」という**関心の分離**に役立ちます。

## 2. 代入のルール（リスコフの置換のイメージ）

**サブクラスのインスタンスは、スーパークラスの変数に代入できる**（is-a なので）。

```java
Animal x = new Dog("ハチ");
```

逆（親の実体を子型変数に無条件代入）は基本できません。必要なら**キャスト**が絡み、安全でない場合があります。

## 3. どのメソッドが呼ばれるか（動的束縛）

メソッド呼び出しは、**実行時の実体の型**に応じて、オーバーライドされた実装が選ばれます（**動的束縛**）。

```java
public void feed(Animal animal) {
    animal.eat();  // 渡された実体が Dog なら Dog の eat
}
```

> **ポイント**
>
> 同じ <code>feed</code> メソッドで、様々な <code>Animal</code> を扱えるため、**if で種類分けが減る**ことがあります（すべてがきちんと <code>eat</code> をオーバーライドしている前提）。

## 4. フィールドは多態性しない（よくある誤解）

**フィールド（インスタンス変数）**は、この文脈では**変数の型側**が使われることがあり、メソッドのように「実体で切り替わる」と考えない方が安全です。多態性で振る舞いを変えたいときは**メソッドをオーバーライド**するのが基本です。

## 5. instanceof とパターンマッチング（入門）

実行時に「本当は何の型か」を調べるには <code>instanceof</code> を使います。

```java
Animal a = new Dog("ポチ");
if (a instanceof Dog) {
    Dog d = (Dog) a;
    System.out.println("犬です");
}
```

Java 16 以降では <code>if (a instanceof Dog d)</code> のように**パターンマッチング**で変数に束縛できます（環境に合わせて調べて使ってください）。

> **注意**
>
> キャスト（<code>(Dog) a</code>）は、**instanceof で確認する**などして誤った型にキャストしないよう注意します。誤ると <code>ClassCastException</code> です。

## 6. 抽象メソッド・抽象クラス（ひとこと）

「親では実装を決めず、子で必ず実装させる」ために <code>abstract</code> メソッドや抽象クラスがあります。多態性の世界をさらに整理する道具として、次のステップで触れるとよいです。

## 7. まとめ

| テーマ | 要点 |
| --- | --- |
| 多態性 | 親型として扱い、実体に応じて振る舞いが変わる |
| 代入 | 子の実体を親型変数に入れられる（is-a） |
| メソッド | オーバーライドが動的束縛で選ばれる |
| フィールド | メソッドほど「その場で切り替わる」とは考えない |
| <code>instanceof</code> | 実行時の型確認。キャストとセットで慎重に |
| 設計 | 呼び出し側を単純にし、拡張は新サブクラスで足せることがある |

> **ポイント**
>
> 多態性は「if の列を減らし、**開放閉鎖原則**に近づける」ための鍵のひとつです。まずは <code>Animal</code> / <code>Dog</code> / <code>Cat</code> のような小さな例を IDE で動かし、どの <code>eat</code> が選ばれるか追ってみてください。
