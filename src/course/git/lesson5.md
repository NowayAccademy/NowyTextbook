# プルリクエストとタグ
PR でレビューし、タグでリリース版を指し示します。

## 本章の目標

本章では以下を目標にして学習します。

- プルリクエスト（PR）が**コードレビューと品質の関所**だと説明できること
- マージ前に確認すべきことの例を挙げられること
- `git tag` で**リリース版**を指すイメージを持てること

## 1. プルリクエスト（Pull Request）

GitHub などで、**ブランチの差分をレビューしてから main に取り込む**ための仕組みです。

- 変更内容の説明（**What / Why**）  
- CI（自動テスト）の結果  
- レビュアからのコメント・承認  

> **ポイント**  
> PR は**学習の場**でもあります。指摘は人格ではなく**コードに向けて**受け取る文化を意識しましょう。

## 2. マージ前の確認例

- 要件を満たしているか  
- テスト・Lint が通るか  
- セキュリティ・パフォーマンスに明らかな問題がないか  

## 3. git tag

リリース版に**人間が読める名前**（例：`v1.2.0`）を付けます。

```bash
git tag -a v1.0.0 -m "First stable release"
git push origin v1.0.0
```

セマンティックバージョニング（major.minor.patch）は現場でよく使われます。

## まとめ

- PR は**レビューと記録**のための窓口である  
- マージ前に**自分で一度チェックリスト**を踏む  
- タグで**リリース地点**を明示する
