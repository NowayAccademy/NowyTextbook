# 履歴の修正と取り消し
`revert` と `reset` の違い、`rebase` の概念を安全側から学びます。

## 本章の目標

本章では以下を目標にして学習します。

- `git revert` が**履歴を残したまま打ち消し**だと説明できること
- `git reset` が**履歴を書き換える危険な操作**になり得ると理解すること
- `rebase` が**履歴をきれいに並べ直す**イメージだと述べられること

## 1. git revert（安全寄り）

**新しいコミット**で「あの変更を打ち消す」差分を作ります。**既に push した履歴**にも使いやすいです。

```bash
git revert <コミットハッシュ>
```

## 2. git reset（注意）

HEAD の位置やインデックスを動かし、**履歴を書き換え**ます。

- `--soft` … コミットだけ取り消し、変更は残る  
- `--mixed` … ステージも外す（デフォルトに近い）  
- `--hard` … **作業内容も破棄**。取り返しが難しい  

> **ポイント**  
> **共有ブランチで `reset --hard` + `push -f` はチームを壊しやすい**です。基本は `revert` を優先しましょう。

## 3. git rebase（概念）

**コミット列を別の土台の上に付け替える**操作です。履歴が一直線にきれいに見える一方、**既に共有された履歴を書き換える rebase** は衝突の元です。

## まとめ

- 公開履歴の取り消しは **`revert` が安全**である  
- `reset --hard` は**自分の手元限定**で慎重に  
- `rebase` は**きれいな履歴**と**チームルール**の両方を意識する
