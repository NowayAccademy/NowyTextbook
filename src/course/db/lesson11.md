# 論理設計：ERD の基本記法
読み書きと、現場での「粒度」の考え方を学びます

## 本章の目標
本章では以下を目標にして学習します。

- ER 図の**エンティティ・属性・リレーション**を説明できること
- 多重度（1対1、1対多、多対多）を図で表現できること
- 「粒度をどこに置くか」が設計の出発点だと説明できること

## 1. ER 図で表すもの

**ERD（Entity Relationship Diagram）**は、データを**どの単位（エンティティ）**で持ち、**どう関連するか**を図示します。

| 要素 | 意味 |
| --- | --- |
| **エンティティ** | 業務上の対象（顧客、注文、商品…） |
| **属性** | エンティティが持つ情報（名前、金額…） |
| **リレーション** | エンティティ間のつながり |

> **ポイント**  
> ERD は「実装のDDL」ではなく、**合意形成と抜け漏れチェック**のための道具です。

## 2. 多重度（カーディナリティ）

- **1対1**：両方が相手に1つだけ対応  
- **1対多**：片方が1、もう片方が複数（例：顧客1・注文多）  
- **多対多**：両方が複数（例：商品とタグ）— 実装では**中間テーブル**が必要になりやすい  

> **ポイント**  
> 多対多をそのまま1本のリレーションで「実装したつもり」になると、後で必ずつまずきます。

## 3. 現場の粒度（どこまでを1行にするか）

例：「住所」を顧客に持つか、別エンティティにするか。  
**更新頻度・履歴の要否・複数住所の有無**で変わります。

> **ポイント**  
> 粒度は「正解1つ」ではなく、**運用とクエリのしやすさのトレードオフ**です。

## 4. 読み方・書き方の練習

- 誰が**主語**か（注文は誰の／どの商品の）  
- 必須か任意か（NULL 許容の議論の種）  
- 識別子（自然キーかサロゲートキーか）  

ツールは **draw.io**、**Mermaid**、専用モデリングツールなど、チームに合わせます。

> **ポイント**  
> 最初は**手書きラフ→レビュー→清書**でも十分です。

## まとめ

- ERD は**エンティティ・属性・リレーション**で世界を切る  
- **多重度**を間違えると、後のDDLとクエリが全部ズレる  
- **粒度**は業務要件と運用から決める
