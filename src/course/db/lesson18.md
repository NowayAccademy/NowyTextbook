# 運用（Ops）：ANALYZE・VACUUM・マイグレーション・バックアップ
PostgreSQL 運用の基本用語と、現場の流れを学びます

## 本章の目標
本章では以下を目標にして学習します。

- **ANALYZE** と **VACUUM** の役割を短く説明できること
- スキーマ変更（マイグレーション）を**安全に進める**ための心構えを持てること
- バックアップ・リストアの**目的**（RPO/RTO）を言葉にできること

## 1. ANALYZE（統計情報）

クエリプランナは**テーブルの統計**を使って計画を選びます。  
統計が古いと、**遅い計画**を選ぶことがあります。

```sql
ANALYZE orders;
```

> **ポイント**  
> 大量ロード後は **ANALYZE** が必要になることが多い。autovacuum の設定は運用ドキュメントで確認。

## 2. VACUUM（死んだ行の整理）

PostgreSQL は更新・削除で**古い行版**が残る仕組みです。  
**VACUUM** はそれを片付け、**ストレージ再利用や可視性**に関わります。

- **VACUUM**：通常運用で定期実行（autovacuum）  
- **VACUUM FULL**：より強い整理だが**ロックが重い**ことがある  

> **ポイント**  
> 「DBが重いから手動 VACUUM FULL」は**判断が難しい**。上級者・DBAと相談するのが無難です。

## 3. マイグレーション運用

スキーマ変更は**アプリと同時リリース**の設計が重要です。

- **後方互換**な変更から入れる（列追加は DEFAULT 付きなど）  
- **二段階リリース**（古いコードが動く状態→切替→掃除）  
- **ロールバック可能か**を先に決める  

> **ポイント**  
> 「DDLを直本番で打つ」は**手順・承認・バックアップ**が整っていないと危険です。

## 4. バックアップとリストア

- **フルバックアップ**＋**WAL（トランザクションログ）**で、ある時点に近い復旧が可能な構成も  
- **RPO**（どれだけデータを失ってよいか）、**RTO**（どれだけで復旧するか）を決める  

リストア手順は**年に1回でも演習**すると、本番障害時に効きます。

> **ポイント**  
> クラウドマネージドDBでも、「**スナップショットが取れている＝目的どおり復旧できる**」とは限りません。復旧テストが重要です。

## まとめ

- **ANALYZE** はプランナのための統計、**VACUUM** は死んだ行の整理である  
- マイグレーションは**互換性とロールバック**を先に設計する  
- バックアップは**RPO/RTO**と**リストア演習**までセットで考える
