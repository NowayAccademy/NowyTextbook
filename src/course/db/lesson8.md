# 日付・時刻と型変換
date / timestamp・CAST・比較と期間抽出を扱います

## 本章の目標
本章では以下を目標にして学習します。

- `date` と `timestamp`（`timestamptz`）の違いをざっくり説明できること
- `CAST` や `::型` での変換を読み書きできること
- 期間抽出で **半開区間** を使う理由を説明できること

## 1. 型のイメージ

| 型 | 意味するもの |
| --- | --- |
| **date** | 日付だけ（時刻なし） |
| **timestamp** | 日付＋時刻（タイムゾーンなしのことが多い） |
| **timestamptz** | 日付＋時刻＋タイムゾーン解釈（PostgreSQL） |

> **ポイント**  
> 「全部 `timestamp` でいい」は短絡的になりがち。**業務上“日”なのか“瞬間”なのか**で選ぶ。

## 2. リテラルと CAST

```sql
SELECT CAST('2025-04-01' AS date);
SELECT TIMESTAMP '2025-04-01 15:00:00';
-- PostgreSQL では次の書き方もある
SELECT '2025-04-01'::date;
```

暗黙の型変換に頼ると、**方言や設定で挙動が変わる**ことがあります。

> **ポイント**  
> アプリからは**パラメータバインド**し、文字列連結で無理やり型を合わせない。

## 3. 比較と「1日」の境界

「2025年4月1日のデータ」を取りたいとき：

- 悪い例：`WHERE ordered_at = '2025-04-01'`（時刻付きだとヒットしない）  
- よい例：**範囲**で指定する  

```sql
WHERE ordered_at >= TIMESTAMP '2025-04-01 00:00:00'
  AND ordered_at <  TIMESTAMP '2025-04-02 00:00:00';
```

これは**半開区間** `[開始, 終了)` で、終日を漏れなく取りやすいです。

> **ポイント**  
> `BETWEEN '2025-04-01' AND '2025-04-01 23:59:59'` は、**ミリ秒の端**で漏れることがあるため推奨されにくいです。

## 4. よく使う関数（PostgreSQL 例）

- `CURRENT_DATE` / `NOW()`：現在日時  
- `date_trunc('month', ts)`：月の切り捨て  
- `age(a, b)`：間隔  

方言差があるため、**プロジェクトのSQL規約**に合わせます。

> **ポイント**  
> `WHERE` に `date_trunc(created_at)` をかけると**インデックスが効きにくい**ことがある。必要なら式インデックスや生成列を検討。

## まとめ

- **日だけ**なら `date`、**瞬間**なら `timestamp` / `timestamptz` を意識する  
- 変換は **`CAST` や明示リテラル**で読みやすく書く  
- 期間抽出は **`>= 開始 AND < 翌境界`** の半開区間が安全なことが多い
