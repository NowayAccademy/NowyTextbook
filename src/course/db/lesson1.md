# データベースとRDBMSの仕組み
テーブル・レコード・カラムの概念とRDBMSの役割を理解します

## 本章の目標

本章では以下を目標にして学習します。

- データベースが必要な理由を説明できること
- RDBMS（リレーショナルデータベース管理システム）とは何かを説明できること
- テーブル・レコード・カラム・スキーマの概念を理解し、身近な例で説明できること

## 1. データベースが必要な理由

### ファイル管理の限界

プログラムでデータを扱う最も原始的な方法は、テキストファイルやCSVファイルに書き込むことです。  
小規模なら問題ありませんが、実際のサービスでは次のような問題が発生します。

**ファイル管理の問題例：**

- 複数のユーザーが同時にファイルを書き換えようとしたとき、データが壊れる
- 特定の条件で検索しようとすると、ファイルを全部読み込んでプログラムで絞り込む必要がある
- 「ユーザー情報」と「注文情報」が別ファイルにあるとき、つなげて見るのが難しい
- バックアップ中に誰かがデータを更新すると、中途半端な状態になる

```
# ファイル管理のイメージ（CSVファイル）
users.csv：
1,田中太郎,tanaka@example.com
2,佐藤花子,sato@example.com

orders.csv：
101,1,2024-01-10,3000
102,2,2024-01-11,1500
```

このCSVで「ユーザーIDが1の人の注文一覧」を取得しようとすると、両方のファイルをプログラムで読み込んで、プログラム側で照合する必要があります。  
これは非常に非効率です。

> **ポイント**  
> データベースは「データを整理して保管し、効率よく検索・更新できる仕組み」です。スプレッドシートに似ていますが、複数人の同時アクセスや大量データに対応できます。

### データの一貫性問題

ファイル管理では「一貫性」の保証が難しいです。

**一貫性の問題例：**
- ユーザーを削除したのに、そのユーザーの注文データが残ってしまう（孤立データ）
- 在庫を減らす処理の途中でサーバーがクラッシュし、注文はされたが在庫は減っていない状態になる
- 同じ商品コードが重複して登録されてしまう

データベースはこれらの問題を「制約」と「トランザクション」という仕組みで解決します。

## 2. RDBMSとは何か

### リレーショナルモデルとは

**RDBMS（Relational Database Management System）** とは、データを「テーブル（表）」の形式で管理し、テーブル間の「関係（リレーション）」を扱えるデータベース管理システムです。

1970年にIBMのエドガー・F・コッドが提唱した「リレーショナルモデル」に基づいています。

### 主なRDBMS製品

| 製品名 | 特徴 | 用途 |
| --- | --- | --- |
| **PostgreSQL** | オープンソース、高機能、本格的な本番環境にも使われる | Webサービス、業務システム |
| **MySQL** | オープンソース、速度重視、Webシステムで広く使われる | Webサービス全般 |
| **SQLite** | ファイル1つで動く軽量DB、サーバー不要 | 小規模アプリ、テスト用 |
| **Oracle Database** | 商用、エンタープライズ向けの高信頼性 | 大企業の基幹システム |
| **Microsoft SQL Server** | 商用、Windows環境との親和性が高い | Windowsベースの業務システム |

> **ポイント**  
> 本コースでは **PostgreSQL** を使用します。オープンソースで無料、かつ本番環境でも広く使われており、学習コストに見合った汎用性があります。

### RDBMSが提供する機能

- **データの永続化**：サーバーを再起動してもデータが消えない
- **同時アクセス制御**：複数ユーザーが同時にアクセスしても整合性を保つ
- **トランザクション**：「全部成功」か「全部失敗」を保証する（後の章で学びます）
- **バックアップ・リカバリ**：障害時のデータ復旧
- **アクセス制御**：ユーザーごとに読み書き権限を設定できる

## 3. テーブル・レコード・カラム・スキーマの概念

### Excelに例えてみる

データベースの概念は、Excelのスプレッドシートに例えるとわかりやすいです。

```
Excelのイメージ：
┌─────────────────────────────────────────────────────────────┐
│ ワークブック（＝データベース）                                   │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ シート「users」（＝テーブル）                              │ │
│  │                                                        │ │
│  │  id  │ name   │  email              │  ← 列（カラム）
│  │ ─────┼────────┼───────────────────  │
│  │   1  │ 田中   │ tanaka@example.com  │  ← 行（レコード）
│  │   2  │ 佐藤   │ sato@example.com    │  ← 行（レコード）
│  └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 各用語の定義

| 用語 | Excelで例えると | 意味 |
| --- | --- | --- |
| **データベース** | ワークブック（.xlsxファイル） | データを管理する入れ物全体 |
| **テーブル** | シート（1枚） | データを格納する表 |
| **カラム（列）** | 列（A列、B列…） | データの属性・項目名 |
| **レコード（行）** | 行（1行、2行…） | 1件分のデータ |
| **スキーマ** | ブックの構造定義 | テーブルの定義・構造 |

### 実際のテーブルのイメージ

```sql
-- usersテーブルのイメージ
-- カラム名: id, name, email, created_at

 id |  name  |       email        |       created_at
----+--------+--------------------+------------------------
  1 | 田中太郎 | tanaka@example.com | 2024-01-10 09:00:00
  2 | 佐藤花子 | sato@example.com   | 2024-01-11 10:30:00
  3 | 鈴木一郎 | suzuki@example.com | 2024-01-12 14:00:00
```

> **ポイント**  
> 「スキーマ」という言葉は文脈によって意味が変わります。「テーブルの定義（どんなカラムがあるか）」を指す場合と、「データベース内の名前空間（テーブルをまとめるフォルダのようなもの）」を指す場合があります。PostgreSQLでは後者の意味でも使われます。

## 4. SQL（Structured Query Language）とは

### SQLの概要

**SQL（エスキューエル）** とは、データベースを操作するための言語です。  
「データを取得して」「データを追加して」「データを更新して」「データを削除して」といった命令をデータベースに伝えるために使います。

### SQLの主な分類

| 分類 | 英語名 | 代表的なコマンド | 役割 |
| --- | --- | --- | --- |
| **DDL** | Data Definition Language | CREATE, ALTER, DROP | テーブルの構造を定義・変更する |
| **DML** | Data Manipulation Language | SELECT, INSERT, UPDATE, DELETE | データを操作する |
| **DCL** | Data Control Language | GRANT, REVOKE | アクセス権限を管理する |
| **TCL** | Transaction Control Language | BEGIN, COMMIT, ROLLBACK | トランザクションを制御する |

### 最も基本的なSQL

```sql
-- データを取得する（SELECT）
SELECT id, name, email
FROM users;

-- データを追加する（INSERT）
INSERT INTO users (name, email)
VALUES ('山田花子', 'yamada@example.com');

-- データを更新する（UPDATE）
UPDATE users
SET email = 'new_tanaka@example.com'
WHERE id = 1;

-- データを削除する（DELETE）
DELETE FROM users
WHERE id = 3;
```

> **ポイント**  
> SQLは「命令形の英語に近い」感覚で読めます。`SELECT name FROM users` は「usersからnameを選べ」という意味です。慣れると英語を読むように理解できるようになります。

## 5. クライアント/サーバーモデルの仕組み

### RDBMSはサーバーとして動く

PostgreSQLはサーバープログラムとして動作します。  
ユーザーやアプリケーションは「クライアント」として、ネットワーク越しにサーバーに接続してSQLを送ります。

```
クライアント/サーバーモデルのイメージ：

┌─────────────────┐           ┌──────────────────────┐
│  クライアント     │           │  DBサーバー           │
│                 │  SQL送信   │                      │
│  ・Webアプリ     │ ────────→ │  PostgreSQL          │
│  ・A5:SQL Mk-2  │           │  （データを管理）       │
│  ・psqlコマンド  │ ←──────── │                      │
│                 │  結果返却  │                      │
└─────────────────┘          └──────────────────────┘
         ↑                            ↑
   あなたのPC                    本番環境ではサーバー上
   （ローカル開発時は同じPCでもOK）
```

### 接続に必要な情報

クライアントがDBサーバーに接続するためには以下の情報が必要です。

| 項目 | 説明 | 例 |
| --- | --- | --- |
| **ホスト（host）** | DBサーバーのアドレス | `localhost`、`db.example.com` |
| **ポート（port）** | DBサーバーが待ち受けるポート番号 | `5432`（PostgreSQLのデフォルト） |
| **データベース名** | 接続先のデータベース | `myapp_db` |
| **ユーザー名** | DBのログインユーザー | `postgres` |
| **パスワード** | ユーザーのパスワード | （設定による） |

> **ポイント**  
> 開発環境では「ホストがlocalhost」つまり自分のPC上でDBサーバーが動いています。本番環境では別のサーバーにDBが置かれることが多いです。

## 6. RDBMSと他のDB（NoSQL）の違い

### NoSQLとは

NoSQL（Not only SQL）とは、リレーショナルモデル以外のデータベースの総称です。  
データの種類や用途によって、RDBMSよりNoSQLが適している場合があります。

| 種類 | 代表製品 | データの形式 | 適したユースケース |
| --- | --- | --- | --- |
| **RDBMS** | PostgreSQL, MySQL | テーブル（行と列） | 一般的な業務システム、整合性が重要な場面 |
| **ドキュメントDB** | MongoDB | JSON形式のドキュメント | 柔軟なスキーマが必要、JSONをそのまま保存したい |
| **キーバリューDB** | Redis | キーと値のペア | セッション管理、キャッシュ、高速読み書き |
| **カラムナーDB** | Cassandra, BigQuery | カラム指向 | 大量データの分析、時系列データ |
| **グラフDB** | Neo4j | ノードとエッジ | SNSの関係図、レコメンデーション |

### RDBMSを選ぶ理由

現場では、以下の理由でRDBMSが選ばれることが多いです。

- **ACID特性**（原子性・一貫性・独立性・耐久性）が保証される
- **複雑な条件での検索・集計**が得意（JOINや集計関数）
- **スキーマ（構造）の強制**によってデータの品質が保たれる
- エンジニアがSQLを知っていることが前提になっている現場が多い

> **ポイント**  
> 「RDBMSかNoSQLか」という議論に正解はありません。本コースではまずRDBMSの基礎をしっかり身につけることを目指します。基礎があれば他のDBへの移行も容易です。

> **現場メモ**  
> 「NoSQLの方がモダンでスケーラブル」というイメージから安易にMongoDBを選び、後から整合性の問題に悩むプロジェクトを見てきました。例えば、注文管理システムをドキュメントDBで作ったところ、「注文ステータスを一括更新する」「特定の条件で集計する」といった処理がRDBMSなら数行のSQLで書けるのに複雑なコードが必要になる、という状況に陥ったケースがあります。業務システムでは「トランザクション・結合・集計が必要」なケースが大半です。「なぜRDBMSでなくNoSQLを選ぶのか」を明確に説明できない場合は、RDBMSを選ぶのが無難です。

## 7. よくある誤解

### 誤解1：「データベース」とは「MySQL」や「PostgreSQL」のことである

**実際：** 「データベース」はデータを格納する仕組み全般を指し、MySQLやPostgreSQLは「RDBMS（データベース管理システム）」の製品名です。  
正確には「PostgreSQLというRDBMSを使ってデータベースを管理している」という関係です。

### 誤解2：SQLはどのDBでも全く同じ文法で動く

**実際：** SQLにはISO標準（標準SQL）がありますが、各RDBMS製品によって方言があります。  
本コースはPostgreSQLの文法を中心に学びます。MySQLやOracle等では一部構文が異なる場合があります。

```sql
-- 例：現在時刻の取得（PostgreSQL）
SELECT NOW();             -- PostgreSQL
SELECT SYSDATE FROM DUAL; -- Oracle
SELECT NOW();             -- MySQL（これはPostgreSQLと同じ）
SELECT GETDATE();         -- SQL Server
```

### 誤解3：「テーブル」と「データベース」は同じ意味

**実際：** 1つのデータベースの中に複数のテーブルが存在します。  
「データベース」は複数のテーブルをまとめる入れ物です。

### 誤解4：データベースがあれば高速になる

**実際：** データベースは適切に設計・運用しないと遅くなります。  
インデックスの設計、クエリの書き方、テーブル設計などが重要です（後の章で学びます）。

## 9. まとめ

| テーマ | 要点 |
| --- | --- |
| データベースの必要性 | ファイル管理ではデータの一貫性・同時アクセス・複雑な検索に対応できない |
| RDBMS | データをテーブル形式で管理するシステム。PostgreSQL、MySQL等がある |
| テーブル・カラム・レコード | Excelのシート・列・行に対応する概念 |
| SQL | データベースを操作するための言語。DDL・DML・DCL・TCLに分類される |
| クライアント/サーバー | アプリやツールがクライアントとして、DBサーバーにSQLを送る仕組み |
| RDBMSの強み | ACID特性・複雑な検索・スキーマによるデータ品質の保証 |
