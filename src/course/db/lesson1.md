# DBコースのオリエンテーション
学習計画表を作り、ゴール・成果物・進め方をそろえます

## 本章の目標
本章では以下を目標にして学習します。

- このコースで「何を・どこまで・どう進めるか」を言語化できること
- ゴール・成果物・週次の進め方を1枚の学習計画表に落とし込めること
- 講師・メンターと認識をそろえるための質問ができること

## 1. なぜ学習計画表が必要か

DB学習は、**用語（RDBMS・正規化・インデックスなど）が多く、手を動かす前に迷いやすい**領域です。  
最初に「到達イメージ」と「提出物」を決めておくと、後半の演習まで一気通貫で進められます。

> **ポイント**  
> 計画表は「縛り」ではなく、**迷ったときに戻る地図**として使います。

## 2. 学習計画表に書く3要素

| 要素 | 書く内容の例 |
| --- | --- |
| **ゴール** | 例：業務テーマのERDが描ける／EXPLAINを読んで改善案を出せる |
| **成果物** | 例：DDL一式、代表SQL、EXPLAINのスクショ、振り返り1ページ |
| **進め方** | 例：週◯は環境＋SELECT基礎、週◯はJOINと集計、など |

### ゴールの立て方（現場目線）

- **スキル**（何ができるか）と**証拠**（何を見せれば分かるか）をセットにする  
- 「SQLが書ける」だけでなく、「**なぜその書き方か**」を説明できるレベルを目安にする

> **ポイント**  
> 成果物は「提出用のきれいな答案」より、**自分が後から見返せるメモ＋実行結果**が実務に近いです。

## 3. 本コースの到達イメージ（参考）

最終的には、次の流れを一人で回せることを目安にします。

1. 要件から**ERD（論理設計）**を起こす  
2. **DDL**でテーブルを定義し、制約・インデックスを意識する  
3. **SQL**で集計・結合・更新まで書き、**EXPLAIN**で遅さの原因を説明する  
4. **運用・権限・バックアップ**の考え方を説明できる

> **ポイント**  
> すべてを完璧にしなくて構いません。**自分のテーマで一連のストーリーを持つ**ことが重要です。

## 4. 学習計画表テンプレート

次の見出しだけ埋めて提出・共有してみましょう。

- **テーマ**（架空のEC、勤怠、予約など）  
- **期間と週あたりの学習時間**  
- **週ごとの予定**（教材のどの章に相当するか）  
- **リスク**（環境で躓きそうな点、SQLで苦手そうな点）  
- **メンターに聞きたいこと**

> **ポイント**  
> テンプレは短くてよいです。**更新しながら育てる**前提で書きます。

## まとめ

- 学習計画表は、**ゴール・成果物・進め方**をそろえるための道具である  
- DB学習は用語が多いので、**早い段階で地図を作る**と効率が上がる  
- 最終的には **ERD→DDL→SQL→EXPLAIN** の一連がイメージできるとよい
