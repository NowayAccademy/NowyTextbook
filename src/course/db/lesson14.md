# 物理設計：テーブル定義書の考え方
桁数・型・固定長/可変長・命名を整理します

## 本章の目標
本章では以下を目標にして学習します。

- テーブル定義書に載せるべき項目を列挙できること
- 数値・文字列・日時の型選定の**判断軸**を持てること
- 固定長と可変長（`CHAR` / `VARCHAR`）の使い分けを説明できること

## 1. テーブル定義書とは

実装前後に共有するための「**契約書**」です。最低限よく書く項目は次のとおりです。

| 項目 | 例 |
| --- | --- |
| 物理名 / 論理名 | `orders` / 注文 |
| カラム名 / 論理名 | `total_amount` / 税込合計金額 |
| 型・桁 | `numeric(12,2)` |
| NULL 可否 | NOT NULL |
| デフォルト | `0` |
| 制約 | PK, FK, UNIQUE, CHECK |
| 説明・備考 | 端数処理ルール、マスク方針など |

> **ポイント**  
> 定義書がないと、**アプリ・バッチ・BI**で解釈がバラつきます。

## 2. 型の選び方（PostgreSQL 前提の例）

| 用途 | 候補 | 注意 |
| --- | --- | --- |
| 金額 | `numeric(p,s)` | `float` は誤差が出るため金額に不向きなことが多い |
| 件数・ID | `bigint` / `integer` | 将来伸びるなら大きめに |
| フラグ | `boolean` または smallint | チーム規約に合わせる |
| 自由記述 | `text` または `varchar(n)` | 上限が業務上決まるなら `varchar` |

> **ポイント**  
> 「とりあえず全部 `varchar(255)`」は運用では楽でも、**検証やインデックス**で後悔することがあります。

## 3. 固定長と可変長

- **`CHAR(n)`**：固定長（PostgreSQL では末尾空白の扱いに注意）  
- **`VARCHAR(n)`**：最大 n 文字までの可変長  

コード値のように**長さが固定**で意味がある場合は固定寄り、名前や備考は可変が一般的です。

> **ポイント**  
> ストレージより、**アプリのバリデーションとDB制約の二重管理**をどう揃えるかが実務の論点です。

## 4. 命名規則

- **スネークケース**（`order_id`）が多い  
- 外部キーは `参照表_id`  
- 日時は `_at`（`created_at`）、日付だけなら `_date`  

チームで**略語の辞書**を持つと迷いが減ります。

> **ポイント**  
> 予約語や方言差（`user` など）に注意し、**クォート必須の名前**は避ける。

## まとめ

- 定義書は**型・NULL・制約・意味**をそろえるためのドキュメントである  
- 金額は **`numeric`** など、誤差の出にくい型を選ぶ  
- 命名は**一貫した規則**より、チーム合意が優先される
