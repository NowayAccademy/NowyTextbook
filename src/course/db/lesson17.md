# パフォーマンス：EXPLAIN とチューニング実務パターン
実行計画の読み方と、CTE・サブクエリの使い分けの入門

## 本章の目標
本章では以下を目標にして学習します。

- `EXPLAIN` / `EXPLAIN ANALYZE` の出力を**ざっくり**読めること
- Seq Scan / Index Scan などの語が何を指すか説明できること
- CTE とサブクエリを、**可読性と最適化**の両面で選ぶ視点を持てること

## 1. EXPLAIN とは

**クエリプランナ**が、どの順序でテーブルにアクセスし、結合するかの**予定**を表示します。  
PostgreSQL では：

```sql
EXPLAIN SELECT ...
EXPLAIN ANALYZE SELECT ...  -- 実際に実行して実測も付ける
```

> **ポイント**  
> `ANALYZE` は本番で重いクエリにそのまま流すと負荷になることがある。**検証環境**で使うのが基本です。

## 2. よく見るノード

| ノード | ざっくり意味 |
| --- | --- |
| Seq Scan | テーブル全表（または広範囲）読み |
| Index Scan | インデックスを使って行を取りに行く |
| Bitmap Index Scan | インデックスで候補を集めてからヒープアクセス |
| Nested Loop | ネストしたループで結合 |
| Hash Join | ハッシュ表を使った結合 |
| Merge Join | ソート済み前提の結合 |

> **ポイント**  
> 「Index Scan だから速い」とは限らず、**ランダムアクセス地獄**になることもあります。`ANALYZE` の**実時間**を見る。

## 3. チューニングの実務パターン

- **`WHERE` に合うインデックス**があるか  
- **想定行数と実行数**がズレていないか（統計 `ANALYZE`）  
- **不要な列 `SELECT *`** を減らす  
- **結合順・結合方法**が想定と違わないか  

> **ポイント**  
> まず**正しい結果**を保ったまま、ボトルネック1か所に集中して直す。

## 4. CTE とサブクエリ（PostgreSQL の注意）

**可読性**では CTE（`WITH`）が優れます。  
一方、バージョンによっては **CTE が「最適化の壁」**として扱われ、サブクエリの方が速い場合があります（環境依存）。

実務では：

- まず**読みやすい形**で書く  
- 重いなら **EXPLAIN 比較**  
- 必要なら **一時テーブル**や**マテビュー**も検討  

> **ポイント**  
> 「CTE が常に遅い／速い」は一概に言えない。**計画と実測**で判断する。

## まとめ

- `EXPLAIN` は**実行計画**、`EXPLAIN ANALYZE` は**実測付き**  
- Seq Scan と Index Scan の違いを押さえ、**実時間と行数**を見る  
- CTE とサブクエリは**可読性と最適化のトレードオフ**があり、計画で検証する
