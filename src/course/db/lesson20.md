# 総合演習：ERD から EXPLAIN まで
後半のまとめとして、一連の流れを自分のテーマで実践します

## 本章の目標
本章では以下を目標にして学習します。

- 小さな業務テーマで **ERD → DDL → データ投入 → クエリ** を一通り行えること
- **遅いクエリを1つ選び、`EXPLAIN ANALYZE` でボトルネックを説明**できること
- インデックスや書き換えで**改善案を1つ以上**出せること

## 1. 課題の流れ（推奨ステップ）

1. **テーマ決め**（例：簡易EC、図書貸出、予約システム）  
2. **ERD 作成**（エンティティ5〜8個程度でよい）  
3. **DDL 作成**（PK/FK、NOT NULL、代表的な CHECK）  
4. **シードデータ投入**（数百〜数千行。合成データでも可）  
5. **業務クエリ**（一覧、集計、結合、更新）を10本程度  
6. **遅いクエリを選定**し、`EXPLAIN ANALYZE` で確認  
7. **改善**（インデックス、条件の見直し、結合順・サブクエリ整理）  
8. **ビフォー・アフター**をメモに残す  

> **ポイント**  
> 完璧な最適化より、**「何がボトルネックか言語化できる」**ことを重視します。

## 2. ERD → DDL のチェックリスト

- 多対多は**中間テーブル**になっているか  
- 金額・数量の型は適切か（`numeric` など）  
- 外部キーは**参照整合**が取れるか  
- 論理削除なら **ユニーク制約**の設計は破綻していないか  

> **ポイント**  
> 最初から全部を正解にしなくてよい。**レビューで直す前提**で書く。

## 3. SQL と EXPLAIN

改善前のクエリ例（意図的に遅くしやすいパターン）：

- 先頭 `%` 付き `LIKE`  
- `WHERE` への関数適用  
- 不要な `SELECT *` と巨大結合  

`EXPLAIN ANALYZE` で **Seq Scan の行数・実時間**に注目します。

> **ポイント**  
> 改善案は必ず**結果行が同じ**ことを `EXCEPT` や件数比較で確認する。

## 4. 提出物の例

- ERD 画像（またはファイル）  
- `schema.sql` / `seed.sql`  
- `queries.sql`  
- 改善メモ（ビフォー・アフターの計画と実時間）  

> **ポイント**  
> ポートフォリオに載せる場合、**個人情報・会社情報が混ざっていない**ことを再確認する。

## まとめ

- 総合演習は **設計→実装→計測→改善** の一連である  
- EXPLAIN は**感覚ではなく根拠**を言うための道具である  
- 提出物は**再現可能**（DDLとデータとクエリ）にまとめる
