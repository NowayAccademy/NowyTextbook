# INNER JOIN と LEFT JOIN
結合条件と、多重化（行が増える罠）を理解します

## 本章の目標
本章では以下を目標にして学習します。

- `INNER JOIN` と `LEFT JOIN` の違いを、**片方の表に行が残るか**で説明できること
- `ON` と `WHERE` の役割の違い（特に外部結合）をイメージできること
- 結合で行数が増える理由と、対策の方向性を説明できること

## 1. INNER JOIN（内部結合）

**両方のテーブルに一致する行だけ**が残ります。

```sql
SELECT o.id, o.ordered_at, c.name
FROM orders o
INNER JOIN customers c ON c.id = o.customer_id;
```

> **ポイント**  
> 「取引のない顧客」は結果に**出てこない**のが INNER の性質です。

## 2. LEFT JOIN（左外部結合）

左表の行は**すべて残し**、右表に合致がなければ右の列は **NULL** になります。

```sql
SELECT c.id, c.name, o.id AS order_id
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id;
```

「全顧客を出し、注文がなければ注文列は NULL」などに使います。

> **ポイント**  
> `WHERE o.id IS NULL` と組み合わせて**未購入顧客**を取るパターンは定番です（外部結合後のフィルタに注意）。

## 3. ON と WHERE（外部結合の落とし穴）

外部結合で、右表の条件を **`WHERE` に書く**と、**INNER と同じ**になってしまうことがあります。  
右表の条件は **`ON` に含める**のが安全なことが多いです。

```sql
-- 例：未キャンセル注文だけを左結合で付けたい
FROM customers c
LEFT JOIN orders o
  ON o.customer_id = c.id
 AND o.status <> 'cancelled'
```

> **ポイント**  
> チームで「外部結合＋WHERE の書き方ルール」を決めると、バグが減ります。

## 4. 多重化の罠

1対多の関係で結合すると、**左の1行が右の複数行に広がる**ため、結果行数が増えます。  
そのまま `SUM` すると**二重計上**になります。

対策の例：

- 先に注文を集計してから顧客と結合する  
- `DISTINCT` を乱用せず、**サブクエリやウィンドウ**で意図を明確にする  

> **ポイント**  
> 「結合後の行数」を**意識的に確認する**（`COUNT(*)` 比較）習慣があるとデバッグが速いです。

## まとめ

- `INNER JOIN` は**一致した行だけ**、`LEFT JOIN` は**左を残す**  
- 外部結合では条件を **`ON` に寄せる**ことが重要なことが多い  
- 1対多結合は**行が増える**。集計前に**重複の影響**を確認する
