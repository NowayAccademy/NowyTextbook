# トランザクションと ACID
一連の処理をまとめて安全に扱う前提を押さえます

## 本章の目標
本章では以下を目標にして学習します。

- トランザクションが「まとまりとして成功か失敗か」を決める単位だと説明できること
- ACID の各要素が現場で何を意味するか、短く言えること
- `COMMIT` / `ROLLBACK` の違いを説明できること

## 1. トランザクションとは

**トランザクション**は、複数の SQL を**ひとまとまり**として扱い、  
**すべて成功**するか、**すべてなかったことにする（ロールバック）**かを決める単位です。

例：口座Aから引き落とし、口座Bへ入金——の2操作は、片方だけ成功してはいけません。

> **ポイント**  
> 「途中まで反映された状態」を防ぐのが、トランザクションの主目的のひとつです。

## 2. ACID 特性

| 特性 | ざっくり意味 | 現場でのイメージ |
| --- | --- | --- |
| **A**tomicity（原子性） | 全部成功か、全部取り消しか | 振込の片側だけ成功、を防ぐ |
| **C**onsistency（一貫性） | ルールを破った状態にならない | 制約・残高ルールを維持 |
| **I**solation（分離性） | 同時実行でもおかしな中間状態を見せない | 他トランザクションの未確定変更をどう見るか |
| **D**urability（永続性） | 確定後は消えない | `COMMIT` 後はクラッシュしても戻らない想定 |

> **ポイント**  
> **分離レベル**や**ロック**は、Isolation をどこまで厳しくするかの設計トピックです。詳細は別章・別資料で深掘りします。

## 3. COMMIT と ROLLBACK

- **`COMMIT`**：このまとまりを確定する。他のセッションからも原則として確定後の状態が見える。  
- **`ROLLBACK`**：このまとまりを取り消し、開始前の状態に戻す。

```sql
BEGIN;
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT;
```

> **ポイント**  
> 対話型クライアントでは、**オートコミット**の有無で挙動が変わります。自分のツールの設定を確認します。

## 4. 現場での「使い方の前提」

- **短く保つ**：トランザクションが長いとロック競合が増える  
- **いらないロックを増やさない**：不要な `SELECT FOR UPDATE` などに注意  
- **冪等性・再実行**：ネットワーク途切れで同じ処理が二重に来る前提の設計もある  

> **ポイント**  
> DB だけでなく、アプリ・メッセージング・バッチの**再試行設計**とセットで考えるのが実務です。

## まとめ

- トランザクションは**まとまりの成功／失敗**を決める単位である  
- ACID は**原子性・一貫性・分離性・永続性**の頭文字  
- **`COMMIT` で確定、`ROLLBACK` で取消**を使い分ける
